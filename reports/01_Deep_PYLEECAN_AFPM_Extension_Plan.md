# Deep PYLEECAN AFPM Extension Plan
## Enabling Native Axial Flux Permanent Magnet Machine Support in PYLEECAN for NeuroFlux

**Version:** 0.1 (Foundation for Implementation)
**Date:** 2026-05-30
**Context:** NeuroFlux Engineering Intelligence Platform (AFPM initial domain). PYLEECAN (Eomys/pyleecan) is the leading open-source Python multiphysics framework for electrical machine design, simulation, and optimization. Current state (audited via GitHub tools): No native AFPM / axial flux / disc / YASA / TORUS / pancake support detectable in code (zero matches for relevant keywords across repo). Strong radial-flux focus with excellent class hierarchy, code generation, FEMM/Elmer interfaces, GUI, and optimization. 

**Strategic Rationale for Extension (vs. Building from Scratch):**
- PYLEECAN already provides mature infrastructure: class system, geometry building, material handling, loss/thermal/structural models, simulation orchestration (FEMM + Elmer), GUI, optimization (genetic, Bayesian), output post-processing, and code generation from CSV definitions.
- Extending it preserves and amplifies existing work (aligns with `agents.md` "preserve work, audit first, integrate OSS").
- NeuroFlux can use PYLEECAN as the authoritative simulation backend for AFPM designs generated by its agents, while adding the missing generative, patent-aware, and multi-fidelity discovery layers on top.
- Community benefit: Contributes back to PYLEECAN ecosystem, increasing adoption of NeuroFlux.
- Feasibility: High, because PYLEECAN's architecture (FrozenClass + generated classes + Methods/ overrides + import_class dynamic loading) is designed for extensibility.

**Core Principle of This Plan:** Rigorous, audit-first, non-destructive. Every step begins with deep understanding of existing code. No deletion or breaking changes without explicit approval and migration path. Prefer extension/subclassing + new modules over modification of core files where possible. Leverage code generator heavily.

## 1. Current PYLEECAN Architecture Audit Summary (Key Findings from Tool Exploration)

### 1.1 Class System & Code Generation (Critical for Extension)
- **Core Mechanism**: Most classes in `pyleecan/Classes/` are **auto-generated** from CSV definitions in `pyleecan/Generator/ClassesRef/`.
  - Example: `Machine.py` header explicitly states: "File generated according to Generator/ClassesRef/Machine/Machine.csv". **WARNING: All changes made in this file will be lost!**
  - There is a `pyleecan/Generator/` directory responsible for this (code generation scripts).
- **Base Classes**:
  - `Machine.py` (abstract): Common properties (frame, shaft, name, desc, type_machine, logger_name). Many methods delegated to `Methods/Machine/Machine/*.py` (build_geometry, check, comp_masses, plot, get_lam_list, comp_periodicity_*, etc.).
  - `MachineSync.py` and `MachineAsync.py` as intermediates.
  - Concrete machines: `MachineIPMSM.py`, `MachineSIPMSM.py`, `MachineLSPM.py`, `MachineSRM.py`, `MachineSyRM.py`, `MachineSCIM.py`, `MachineDFIM.py`, `MachineWRSM.py`, `MachineUD.py` (user-defined), etc. All inherit and specialize.
- **Registration**: `pyleecan/Classes/Class_Dict.json` and `import_all.py` + dynamic `import_class` function enable runtime loading by string name.
- **Lamination Core**: `Lamination.py` (and subclasses like `LamSlot.py`, `LamSlotWind.py`, `LamSlotMag.py`, `LamHole.py`, etc.) handle stator/rotor geometry, materials, ventilation, notches, holes, slots, windings, magnets. This is where much AFPM-specific geometry will live or be extended.
- **Geometry Primitives**: Rich set (`Surface`, `SurfLine`, `SurfRing`, `Slot*` (many variants W10–W63, M*, UD), `Hole*`, `Magnet`, `Winding`, `Arc*`, `Segment`, `Trapeze`, etc.). AFPM will require new or extended primitives for **disc rotors**, **axial magnet placement**, **yokeless segmented stators (YASA)**, **flat axial airgaps**, and **3D flux considerations** (currently mostly 2D radial assumptions).
- **Simulation**: 
  - `MagFEMM.py` / `OutMagFEMM.py` (2D primarily, widely used).
  - `MagElmer.py` / `StructElmer.py` / `ElmerResults*` (better multi-physics and some 3D potential).
  - `Simulation.py`, `Simu1.py`, `VarSimu.py`, `VarOpti.py` for orchestration and optimization.
- **Other Key**: `Output.py` / `Out*` hierarchy for results; `Material.py` + magnetic/electrical/structural/HT models; `Loss*` models (Bertotti, Joule, Magnet, etc. — some may need AFPM-specific eddy current enhancements for disc magnets/rotors); GUI components.

**Conclusion from Audit**: PYLEECAN is **not AFPM-ready** today. The radial-flux-centric assumptions (cylindrical coordinates, radial airgap length, typical slot/hole patterns for radial machines) dominate. However, the modular, generated-class architecture + existing Elmer support makes a clean, maintainable extension highly feasible without forking.

### 1.2 Gaps for AFPM
- No machine classes or topology flags for axial flux configurations (SSSR, DSSR, SSDR/TORUS, YASA, coreless, multi-disc, bearingless AFPM, etc.).
- Geometry builders assume radial flux paths and cylindrical symmetry; lack disc rotor models, axial magnet arrays (surface-mounted on flat discs, Halbach axial, segmented), yokeless armatures, axial force computation as first-class output.
- Airgap and periodicity calculations are radial-oriented.
- GUI and tutorials lack AFPM examples or topology selection.
- Some loss/force/thermal models may need AFPM-specific corrections (e.g., larger end-winding effects in axial, magnet eddy currents in disc geometries under axial flux, structural deflection under axial magnetic pull affecting airgap).
- 3D simulation workflows are present via Elmer but not streamlined or validated specifically for AFPM benchmark cases.

## 2. High-Level Extension Strategy

**Recommended Approach: Hybrid — Code Generation + Targeted Manual Implementation + NeuroFlux Hooks**

1. **Extend the Generator** (preferred long-term): Add new CSV definitions under `Generator/ClassesRef/Machine/` and `Generator/ClassesRef/Lamination/` (or new Axial-specific subdirs) for AFPM variants. Regenerate classes.
2. **New/Extended Geometry Layer**: Create `pyleecan/Classes/` entries for axial-specific components (e.g., `RotorDisc.py`, `StatorAxial.py`, `MagnetAxial.py`, `SlotAxial*` or reuse/extend `SlotUD`/`LamSlot` with axial flags). Implement `build_geometry` for disc topologies.
3. **New Machine Classes**: `MachineAFPM.py` (base) + concrete `MachineAFPM_DSSR.py`, `MachineAFPM_YASA.py`, `MachineAFPM_Coreless.py`, etc. (or a single `MachineAFPM` with a `topology` enum/parameter for flexibility — decision to be validated in audit).
4. **AFPM-Specific Methods**: Implement in `Methods/Machine/MachineAFPM/` (or per subclass) for:
   - `build_geometry` (discs, magnets on faces, axial stators).
   - `comp_axial_force`, `comp_airgap_flux_density_axial`, `comp_torque_axial`, `comp_back_emf_axial`.
   - Periodicity and symmetry adapted to axial (often different pole/slot combos and 3D periodicity).
   - Integration with existing `comp_masses`, `check`, `plot` (extend 3D plotting).
5. **Simulation Enhancements**: 
   - Ensure/ extend `MagElmer` workflows for full 3D AFPM (or quasi-3D approximations validated against literature).
   - Add AFPM-specific post-processing in `OutMag*`.
   - Multi-fidelity: Analytical magnetic circuit (quick sizing) → 2D FEMM (where applicable) → 3D Elmer.
6. **GUI & Usability**: Add AFPM topology selector, parametric widgets for disc dimensions (Do, Di, axial lengths, magnet thickness, airgap), 3D preview hooks.
7. **NeuroFlux Integration Layer** (new, non-intrusive):
   - Add optional hooks or a `neuroflux/` subpackage or extension points (e.g., `Machine.get_neuroflux_params()`, export to NeuroFlux parametric dict, accept geometry from NeuroFlux agents).
   - Allow NeuroFlux to drive PYLEECAN simulations programmatically with full provenance.
   - Expose AFPM machines to NeuroFlux knowledge graph and generative agents.
8. **Testing, Validation, Documentation**: New test cases against published AFPM benchmarks (torque, efficiency, forces from literature). Tutorials for common AFPM use cases (direct-drive wind generator). Update docs.

**Phased Rollout (Aligned with NeuroFlux Roadmap & User's Rigorous Process)**:
- **Phase 0 (Current — Audit Complete)**: This document. Full understanding of generator, class loading, geometry pipeline, simulation interfaces.
- **Phase 1 (Foundation — 2–4 weeks)**: Extend Generator with minimal AFPM CSV defs. Create base `MachineAFPM` skeleton + one concrete topology (e.g., DSSR). Implement core `build_geometry` + basic analytical sizing. Update Class_Dict/import. Non-breaking (new classes only).
- **Phase 2 (Core Physics — 4–6 weeks)**: AFPM-specific methods for flux, torque, forces, losses (leveraging/extending existing Loss models). Basic FEMM/Elmer integration validation on reference geometry. GUI prototype for topology selection.
- **Phase 3 (Advanced & 3D — 4–8 weeks)**: Full YASA/coreless/multi-disc support. Enhanced 3D Elmer workflows + validation vs. literature prototypes. Structural/thermal coupling for axial forces and magnet temp. Optimization examples.
- **Phase 4 (NeuroFlux Integration & Polish — ongoing)**: Hooks for generative design, surrogate model training data export, patent-aware design metadata, full documentation/tutorials, contribution back to upstream PYLEECAN (if desired).
- **Phase 5 (Generalization)**: Lessons applied to other domains (motors, etc.).

**Risks & Mitigations**:
- **Generator Complexity**: Audit Generator scripts thoroughly first. Start with manual class creation for speed in Phase 1, migrate to generator-driven in Phase 2+.
- **3D Accuracy & Compute**: AFPM often needs 3D. Mitigate with validated quasi-3D models + clear user guidance on when full 3D Elmer is required. Leverage existing Elmer support.
- **Breaking Changes**: Strictly additive. Use subclassing and new modules. Maintain full backward compatibility for radial machines.
- **Validation Data**: Use open literature benchmarks + future hardware prototypes (tie to your test capabilities). Quantify model fidelity.
- **Scope Creep**: Prioritize DSSR + YASA first (most common high-performance generator topologies). Defer exotic variants (bearingless, etc.) to later phases.

## 3. Detailed Technical Implementation Outline

### 3.1 Generator & Class Creation (Phase 1 Priority)
1. **Audit Generator**:
   - Explore `pyleecan/Generator/` fully (scripts, templates, how CSV → .py works, how Methods are linked).
   - Understand `ClassesRef/` structure (Machine/, Lamination/, etc.).
2. **New Definitions**:
   - Create `Generator/ClassesRef/Machine/MachineAFPM.csv` (base properties + topology selector, rotor_disc, stator_axial, etc.).
   - Create or extend lamination CSVs for axial geometries (e.g., `LamDisc.csv` or parameters in existing Lam* for axial mounting).
   - Possibly new primitive CSVs if needed (e.g., `DiscRotorSurface`, `AxialMagnet`).
3. **Regenerate or Manually Bootstrap**:
   - Run generator to produce `MachineAFPM.py` skeleton.
   - Or (faster initial) manually create `MachineAFPM.py` following exact pattern of e.g. `MachineIPMSM.py` (inherit from MachineSync, set type_machine, implement/override key methods).
4. **Registration**:
   - Update `Classes/import_all.py` and ensure `Class_Dict.json` includes new classes (or it auto-updates).
   - Test dynamic loading: `import_class("pyleecan.Classes", "MachineAFPM")`.

### 3.2 Geometry & Core Methods (Phases 1–2)
- **New/Extended Classes** (examples):
  - `RotorDisc.py` (inherits/extends Lamination or new base): Flat disc geometry, magnet pockets/arrays on axial faces, Halbach option, structural considerations for centrifugal + axial pull.
  - `StatorAxial.py` or extensions to `LamSlotWind`: Support for yokeless segmented (YASA-style), tooth-coil windings optimized for axial, open/closed slots in axial context.
  - `MagnetAxial.py` (extends Magnet): Orientation, segmentation for eddy reduction in axial flux, temperature-dependent properties in disc context.
- **Key Methods to Implement/Override** (in `Methods/Machine/MachineAFPM/` or specific topology):
  - `build_geometry`: Construct full 3D-capable surfaces for discs, magnets, stators, airgap (axial direction primary).
  - `comp_axial_magnetic_force` (critical — often dominant in AFPM; existing force methods are mostly radial).
  - `comp_airgap_flux` / `comp_B_emf` adapted for axial main flux path.
  - `comp_periodicity` (spatial/time) — AFPM often has different symmetries.
  - `comp_masses`, `comp_length_airgap_active` (axial length becomes critical dimension), `get_lam_list` (handle disc "laminations").
  - `plot` / `plot_anim` extensions for 3D or exploded axial views (may leverage existing or add VTK/Plotly hooks).
- **Reuse Extensively**: Many `Slot*`, `Hole*`, `Winding*`, `LossModel*`, `Mat*` can be reused or lightly parameterized with an `is_axial` flag or new axial-specific variants.

### 3.3 Simulation & Output (Phases 2–3)
- Validate/extend `MagFEMM` for AFPM where 2D approximations suffice (some quasi-3D models exist in literature).
- Prioritize `MagElmer` + `StructElmer` for true multi-physics (EM + thermal + mechanical deflection under axial force — airgap closure risk).
- New or extended `OutMagAFPM.py` or methods in `OutMag.py` for axial-specific post-processing (force vectors, disc-specific loss maps, efficiency including axial effects).
- Add AFPM benchmark validation scripts (compare to published papers on torque density, force, efficiency for DSSR/YASA reference machines).

### 3.4 GUI, Optimization, Documentation (Phases 2–4)
- GUI: New topology selector in machine creation wizard; parametric panels for Do/Di, axial thicknesses, magnet count/arrangement, winding type suited to axial.
- Optimization: Ensure `VarOpti` / `Opti*` work seamlessly with new AFPM classes (objectives like torque density, axial force minimization, efficiency @ specific speed).
- Docs/Tutorials: New section "Axial Flux Machines" with step-by-step DSSR generator example, comparison to radial, manufacturing notes (tie to MAGGEN/OpenAFPM insights).
- NeuroFlux Hooks (new module, e.g., `pyleecan/NeuroFlux/` or contribution to NeuroFlux repo):
  - `Machine.to_neuroflux_dict()`: Export full parametric description + materials + simulation results for knowledge graph / generative agents.
  - `from_neuroflux_geometry(geom_dict)`: Accept parametric output from NeuroFlux geometry agent and instantiate PYLEECAN machine.
  - Simulation runner wrapper that logs full provenance for NeuroFlux audit trail.
  - Optional: Expose PYLEECAN as a tool/agent in NeuroFlux multi-agent system.

### 3.5 Testing & Quality
- Unit tests for new classes/methods (geometry correctness, force calculation vs. analytical, periodicity).
- Integration tests: Full simulation workflow on reference AFPM geometry → compare key metrics (average torque, ripple, axial force, efficiency) to literature or commercial tools.
- Regression: Ensure all existing radial machines unaffected.
- Performance: Profile new 3D paths; provide guidance on when to use reduced-order vs. full 3D.
